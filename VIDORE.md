# ViDoRe V3 Benchmark

**Paper:** *ViDoRe V3: A Comprehensive Evaluation of Retrieval Augmented Generation in Complex Real-World Scenarios*
**Authors:** Loison, Macé, Edy, Xing, Balough, Moreira, Liu, Faysse, Hudelot, Viaud (Illuin Technology, NVIDIA, CentraleSupélec)
**arXiv:** 2601.08620v1 (January 2026)
**HuggingFace:** https://hf.co/vidore

---

## Overview

ViDoRe V3 is a multimodal RAG benchmark for evaluating end-to-end retrieval pipelines over visually rich PDF documents. It jointly evaluates three RAG components:
1. **Retrieval** — page-level NDCG@10
2. **Answer generation** — pass@1 judged by LLM
3. **Visual grounding** — bounding box localization

Unlike prior benchmarks (DocVQA, ViDoRe V1&V2), it evaluates all three components together rather than in isolation, and targets multi-page, multi-document contexts rather than single-page comprehension.

## Query Taxonomy

Each query is defined along two orthogonal axes:

**Query Type** (7 types, defining information need):
- `open-ended`, `extractive`, `numerical`, `multi-hop`, `compare-contrast`, `boolean`, `enumerative`

**Query Format** (3 types, defining syntactic structure):
- `question`, `keyword`, `instruction`

Extractive queries are most frequent (annotator preference). Multi-hop queries are the rarest and hardest to scale. Question format consistently outperforms Instruction and Keyword formats across retrievers.

## Annotation Process

### Query Generation (3 streams)
1. **Human-page queries** — annotators write queries directly from raw PDF pages
2. **Human-summary queries** — annotators write queries from LLM-generated section summaries (reduces extractive bias)
3. **Synthetic-summary queries** — generated by NeMo Data Designer (Qwen3-235B), filtered by LLM-as-judge

### Query-Page Linking
- Qwen2.5-VL-32B pre-filters candidate pages (flags relevance per page)
- Queries spanning >30 flagged pages are discarded
- 1–3 human annotators rate remaining pages on a 3-point scale: Not Relevant / Critically Relevant / Fully Relevant
- Gwet's AC2 inter-annotator agreement: **0.760**

### Answer Generation
- Human annotators write answers based on the pages they marked relevant
- Qwen2.5-VL-32B synthesizes a single final golden answer from all annotator responses

### Bounding Box Annotation
- Annotators delineate bounding boxes around supporting evidence
- Each box is attributed a modality type: Text, Table, Chart, Infographic, Image, Mixed, Other
- Inter-annotator IoU: **0.50**, F1: **0.60**

## Evaluation Metrics

### Retrieval: NDCG@10
- Graded relevance: 0 (not relevant), 1 (critically relevant), 2 (fully relevant)
- Reported on 0–100 scale
- Macro-average over all datasets (following MTEB conventions)

### Answer Generation: pass@1
- Binary correct/incorrect per query, judged by LLM
- A query is **easy** if any model in a 6-LLM panel answers it correctly *without* retrieval context — 48.6% of queries fall in this category
- A query is **hard** otherwise
- Results are reported globally and stratified by easy/hard

### Visual Grounding
- LLM predicts bounding boxes in XML format inline with the answer
- Evaluated against human-annotated boxes using zone-based IoU and F1

### Best Textual Retrievers (cross-lingual avg)

|---|---|---|
| Qwen3-8B | 8 | 51.0 |

### Best Visual Retrievers (cross-lingual avg)
| Model | Size (B) | Avg |
|---|---|---|
| ColEmbed-3B-v2 | 3 | **59.8** |
| Jina-v4 (visual) | 3 | 57.6 |
| ColNomic-7B | 7 | 57.4 |

Visual retrievers consistently outperform textual retrievers of equivalent size.

### Effect of Reranking (Table 2, Jina-v4)
| Pipeline | Avg |
|---|---|
| Jina-v4 (text) | 50.4 |
| Jina-v4 + zerank-2 | **63.6** (+13.2) |
| Jina-v4 (visual) | 57.6 |
| Jina-v4 + jina-reranker-m0 | 57.8 (+0.2) |

**Key finding:** textual rerankers yield large gains (+13.2 pts); visual rerankers are nearly ineffective (+0.2 pts avg, degrades on 4/8 datasets).

## Key Generation Results (pass@1, Table 3, Gemini 3 Pro)

| Pipeline | Modality | CS | Avg Hard | Avg Global |
|---|---|---|---|---|
| Oracle (text) | Text | 80.9 | 62.3 | 70.6 |
| Oracle (image) | Image | 86.5 | 68.2 | 72.6 |
| Jina-v4 + zerank-2 | Text | 80.9 | 52.1 | 64.9 |
| Jina-v4 + zerank-2 + ColEmbed | Hybrid | 86.5 | 54.7 | 65.7 |
| ColEmbed-3B-v2 | Image | 82.3 | 51.7 | 62.7 |

**Hybrid configuration:** top-5 images (ColEmbed) concatenated with top-5 text pages (Jina+zerank-2), no deduplication.

### Key findings
- **Visual context helps on hard queries** — image context outperforms text by 2.4–2.8 pts on the hard subset
- **Hybrid is best on hard queries** — 54.7% vs 52.1% (text) and 54.5% (image)
- **Easy queries inflate scores** — 48.6% of queries are answerable from parametric knowledge alone; controlling for this is critical for valid comparison
- **10-point gap remains** between best non-oracle (54.7%) and image oracle (64.7%) on hard queries — retrieval is still the bottleneck

## Important Implementation Notes

### Text Extraction for Textual Retrieval
- The paper uses **NeMo Retriever extraction** (NVIDIA) as the text source for Jina-v4
- Footnote 3 (Section 4.1): chunking within pages or adding image descriptions did **not** improve retrieval — the simplest pipeline (plain markdown, no chunking) performed best
- This means retrieval benefits from **concise, dense text** rather than exhaustive transcription

### Dataset Split Strategy
- 8 public datasets released for open research
- 2 datasets (Nuclear, Telecom) kept private as hold-out sets to prevent overfitting
- Performance on public sets may overestimate real-world generalization

### Cross-lingual Evaluation
- Retrieval degrades 2–3 pts in cross-lingual settings (query language ≠ document language)
- English monolingual evaluation scores are systematically higher than the cross-lingual averages reported as primary results

### HuggingFace Dataset Format
Each subset (`vidore/vidore_v3_{subset}`) provides:
- **Corpus**: `corpus_id` (int, 0 to N-1), `image` (PIL), `markdown` (NeMo extracted text), PDF metadata
- **Queries**: `query_id` (int, globally unique across subsets), `query` (str), `answer` (str)
- **Qrels**: `query_id`, `corpus_id`, `score` (0/1/2)
