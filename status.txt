## Key cross-subset observation (major finding)

The best-condition pattern SHIFTS between English and French:

  English (avg over 3 subsets): hybrid_deepseek 88.7  ← best
  Physics FR:                   jina_nemo_reranked 92.1  ← best
  Finance_fr:                   jina_nemo_reranked 80.9  ← best

The midway-report conclusion ("DeepSeek's faithful OCR helps most in hybrid
mode when combined with page images") does NOT generalize to French. On both
French subsets, reranked text-only dominates hybrid. Mechanisms implicated:

- The hybrid_deepseek > hybrid_nemo advantage is corpus-dependent:
    EN avg:      +1.8 pts  (DeepSeek better)
    Physics FR:  +2.9 pts  (DeepSeek better — pattern holds but reranked
                 text-only still wins overall)
    Finance_fr:  -0.6 pts  (DeepSeek WORSE — the hybrid advantage flips)
  DeepSeek's faithful-OCR edge becomes invisible or negative on French
  corporate filings, likely because NeMo's extraction of French tables
  and exhibits is strong enough to close the gap.
- Visual retrieval relatively weaker on text/table-heavy French documents:
  colembed on finance_fr = 70.3 (vs 83.7 EN avg, 83.1 physics FR).
  Finance_fr content is mostly tables, exhibits, numbers — text retrieval
  has more to work with than visual signals.
- Reranker is far more impactful on finance_fr than physics FR:
  NeMo: +19.3 (fin_fr) vs +5.0 (phys_fr); DeepSeek: +16.9 vs +6.3.
  Finance_fr corpus has many structurally similar boilerplate pages
  (tables, exhibits) that only a reranker can disambiguate.
- Retrieval value varies wildly by subset, proportional to how document-specific
  the queries are:
    computer_science EN: +3.7 pts  (closed 91.6 → best 95.3)
    physics FR:          +4.3 pts  (closed 87.8 → best 92.1)
    pharmaceuticals EN:  +20.1 pts (closed 68.4 → best 88.5)
    finance_en EN:       +27.5 pts (closed 56.0 → best 83.5)
    finance_fr FR:       +40.3 pts (closed 40.6 → best 80.9)
  Physics and CS are dominated by parametric knowledge (conceptual/textbook
  queries the model knows). Finance subsets test document-specific facts
  (exact $ amounts, subsidiary names) the model can't recall — retrieval
  is where the win lives.

## What did not work

- **Multimodal OCR prompt** — changed DeepSeek prompt to "Extract all text. For each figure or
  chart, describe its content in detail as part of the text flow." Did not improve NDCG@10;
  reverted to the default `<|grounding|>Convert the document to markdown.` prompt.
- **DeepSeek markdown cleaner** — stripped image placeholders (![](images/*.jpg)) and HTML table
  tags before embedding. No measurable impact on NDCG@10; removed.
- **DeepSeek-OCR-2 as text extractor** — overall NDCG@10 avg 55.82 (no rerank) and 71.02
  (zerank-2) vs NeMo 57.97 / 75.09. Gap widens after reranking (−2.15 → −4.07 pts).
  OCR output is noisier and longer than NeMo; does not help retrieval quality on these subsets.

## Mitigating DeepSeek-OCR-2's blindness to figures/charts

DeepSeek outputs image placeholders for visual content, so figures are invisible to the text stream.
The generator can compensate at answer-generation time:

**Primary (already in pipeline design)**
- Send the top-5 retrieved page *images* (from ColEmbed) alongside the top-5 markdown texts to the
  generator VLM. The model sees the actual figures directly — no OCR needed for visual content.
  This is the ViDoRe v3 hybrid protocol and is already planned.

**Alternatives if the hybrid context is insufficient**
- Figure captioning: DeepSeek already extracts figure images to disk (images/0.jpg, etc.) alongside
  the markdown. Run a captioning model (BLIP-2, LLaVA, or the generator VLM itself) on those
  saved images and replace the placeholders with the generated captions before embedding. Targeted
  and avoids re-running OCR.
- Multimodal OCR prompt: ask DeepSeek (or another model) to describe figures inline as part of the
  markdown. We tried this (prompt: "describe each figure in detail as part of the text flow") but
  reverted because it did not improve NDCG@10. May still help answer accuracy — worth revisiting
  after end-to-end QA is implemented.
